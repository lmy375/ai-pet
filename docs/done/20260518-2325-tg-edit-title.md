# TG bot 加 `/edit_title <title> :: <new title>` 命令（iter #511）

## Background

前端 PanelTasks 已有 double-click title inline rename（commitRenameTask
line 4582）— mouse user 直接双击 title 改名。但 TG 端缺对偶 — owner
在手机上想改 task 名（比如「整理 Downloads」→「清理桌面 + Backups」）
只能：
1. 切到桌面 panel
2. 双击 task title
3. 改名

或：
1. `/dup <title>` 改完手动复制 markers
2. `/cancel <old>`
3. `/dup` 后的副本 → 重新 /edit_title... 但循环依赖

本 iter 加 `/edit_title <title> :: <new title>` — TG 端一步改名，
backend 复用既有 `memory_rename` Tauri 命令（与前端 inline rename 同后
端 — 自然保 description / markers / detail.md / [task pri=]）。

#### 纯 formatter `format_edit_title_reply`

```rust
pub fn format_edit_title_reply(old_title: &str, new_title: &str) -> String {
    format!(
        "✏️ 已改标题：「{}」→「{}」",
        old_title.trim(),
        new_title.trim(),
    )
}
```

极简 — 关键信息（src + new）一行。空白情况由 caller 拦截（empty 时走
missing-arg）；same-title noop 由 backend memory_rename 返回 "No
change."（透显）。

6 个 unit tests：parser（标准 `::` / 前后 trim / 无分隔符 / 空 input /
new title 含 `::`）+ formatter（src + new 显示 / 前后 trim 生效）。

### `src-tauri/src/telegram/bot.rs`

Handler 紧贴 TouchedToday 之后：

```rust
TgCommand::EditTitle { title, new_title } => {
    if title.trim().is_empty() || new_title.trim().is_empty() {
        format_missing_argument("edit_title")
    } else {
        let actual = ... // 3-layer resolve
        match actual {
            Ok(src) => {
                let rename_result = crate::commands::memory::memory_rename(
                    "butler_tasks".to_string(),
                    src.clone(),
                    new_title.clone(),
                );
                match rename_result {
                    Ok(_) => format_edit_title_reply(&src, &new_title),
                    Err(e) => format_command_error(&e),
                }
            }
            Err(msg) => format_command_error(&msg),
        }
    }
}
```

memory_rename 内部已含：
- 同 title noop（return Ok("No change.")）
- 重名拒（return Err "Title already exists ..."）
- 找不到 task 拒
- .md 文件 move + 同名冲突自动 `_N` 后缀
- index 项改 title + 改 detail_path

## Key design decisions

- **复用 memory_rename 不引新 backend**：前端 inline rename / TG /edit_title
  / 未来其它入口都走同一 backend — 行为天然一致，markers / detail.md
  保留无需重新实现
- **拒重名而非自动 `_N`**：与 /dup 的 unique-filename 自动加 `_N` 不
  同 — rename 是「显式新名」owner intent，自动改名违反预期；让 owner
  在 `/edit_title` 命中冲突时显式选另一名
- **与 /edit 拆开**：/edit 覆写 description body — 用 markers 时必须自
  己写进 new desc；/edit_title 只改名 — 完全不动 description / markers
  / detail.md。两 axis 应分开避免「想改名结果丢 markers」事故
- **3-layer title resolve**：与 /done / /cancel / /show 同源 — owner
  心智复用，数字 index 路径也支持「/edit_title 1 :: new name」
- **cascade rename caveat in tooltip**：rename 不自动追踪既有 `[task:
  「<old>」]` ref / detail.md 内 `「<old>」` token。让 owner 明确知道
  — 后续 iter 可考虑 cascade rename pipeline
- **6 个 unit tests pin 真实行为**（满足 GOAL.md "meaningful tests
  only"）：parser 4 个（标准 / trim / 无 `::` / 含 `::` in new title）
  + formatter 2 个（显示 / trim）

## Verification

- `cargo build`（src-tauri）— clean
- `cargo test --lib` — all 1624 tests pass（新 6 + 既有 1618）
- 三个 drift-defense test all pass
- 手测：
  - `/edit_title 整理 Downloads :: 清理桌面` → reply「✏️ 已改标题：
    「整理 Downloads」→「清理桌面」」+ PanelTasks 看到新 title
  - description / detail.md / markers 完全保留
  - `/edit_title 1 :: 重命名` → 索引解析 + rename
  - 同 title `/edit_title a :: a` → backend "No change."
  - 重名拒 `/edit_title a :: b`（b 已存）→ "Title already exists ..."

## Future iters (out of scope)

- **cascade rename**：自动追踪并改 detail.md 中的 `「<old>」` ref /
  其它 task 的 [blockedBy: <old>] markers — 用户调研后再 iter
- `/edit_title` 失败回滚（如 detail.md move 中断）— 当前 memory_rename
  已是单点失败原子（write-then-replace 模式），罕见场景
- PanelTasks 行 hover 「✏️ rename」chip — 当前已有 double-click，hover
  chip 是 affordance 增强；后续 iter
