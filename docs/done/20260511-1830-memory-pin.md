# PanelMemory 单条记忆 pin 置顶

## 需求

高价值记忆（user_profile 里"用户偏好 dark theme"、todo 里"每周三看医生"等）
混在长列表底部，每次切到 memory tab 都要滚 / 搜才找到。加 📌 / 📍 按钮：
点一下把该 item pin 到 category 顶部，再点一下取消。

## 实现

`src/components/panel/PanelMemory.tsx`：

- 新 state `pinnedKeys: Set<string>`，初值从 `localStorage["pet-memory-pinned"]`
  懒加载（JSON 数组转 Set）。key = `${catKey}::${title}`，跨类目同名不冲突。
- `togglePin(catKey, title)`：写新 Set + JSON 同步进 localStorage；私密浏览
  / 容量满 try/catch 吞掉，UI state 仍生效。
- category 渲染前两段拼接排序（保留原顺序）：`pinned.push / rest.push` 二
  分，再 `[...pinned, ...rest]`。优于 `.sort` comparator —— 显式表达"两段
  拼接"而非依赖 stable sort 假设。`pinned.length === 0` 时直接返回 cat.items
  避免无谓数组分配。
- 折叠规则（> 10 条折叠到前 5）改用 `sortedItems.length` 而非
  `cat.items.length`，保证 pin 项必落在 preview 段（先 5 条本来就是首 5）。
- 按钮放在每条 item 编辑 / 删除按钮组的最前：pinned 时变黄底深棕字 + 加粗
  + 实心 📌；未 pin 时灰底 📍（lightweight 视觉差，hover tooltip 解释切换
  语义）。

## 为什么不动 backend

memory 文件 frontmatter 里加 `pinned: true` 会让 LLM 在 prompt 里也看到该
字段，可能误以为是语义信息。pin 是用户的"UI 偏好"——只影响 panel 内看到
的顺序，不应改变模型可见的 memory body。localStorage per-user 也避开了多
设备同步问题（GOAL.md 明确不做云同步）。

## 验证

- `npx tsc --noEmit` clean
- 行为：
  - 点某条 📍 → 立刻跳到 category 顶 + 按钮变 📌（黄）
  - 再点 → 回原位
  - 刷面板 / 关重开 panel → pin 状态恢复
  - 折叠（> 10 条）态下 pin 必在 preview 段（前 5 条）
  - 重命名 / 删除该条 memory → pin key 变 dangling，下次 reload list 自动
    被忽略（不命中 cat.items），无 UI 异常

## 不在本轮范围

- 没做"按 pin 时间排序"——同一 pin 区内保留 backend 原顺序（updated_at 升）
  足够，多排序维度反而混乱
- 没做 pin 上限：理论上用户能 pin 所有 item 让 pin 变 noop，但概率低 + 自
  纠正（取消即可）
- 没在 PanelMemory 头部做"全局 pin 视图"（跨 category 的 pin 项聚集）：与
  category 维度互补但 layout 改动大；先看 user 是否需要
- 没在搜索结果列表用 pin（searchResults 是 flatten 结果，pin 排序在那里意
  义不大）—— 搜索本身已经按关键字相关性筛过
