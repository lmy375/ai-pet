# TG bot 加 `/dup <title>` 命令（iter #497）

## Background

「以这条 task 为模板建一条相似的」场景 owner 在 TG 端只能：
1. `/show <title>` 看 raw description
2. 手动复制 raw text
3. 手动剥 [done] / [snooze:] / [result:] 等终态 markers
4. `/task <new title>` 重输

四步 + 错漏（容易忘剥 markers 或忘填 priority）。

本 iter 加 `/dup <title>` — 一键复制 task 为新 pending 实例：
- 继承 schedule / pinned / silent / blockedBy / reminderMin / tags / pri / due
- 剥 done / result / error / cancelled / snooze / archived / origin:tg
- title 自动加 `(副本)` 后缀；同源多次 dup 由 memory_edit unique-filename
  兜底自动加 `_1` / `_2`

## Changes

### `src-tauri/src/task_queue.rs`

新增 pub helper `strip_for_dup`（line ~270）：

```rust
pub fn strip_for_dup(description: &str) -> String {
    let cleaned = remove_bracketed_segments(
        description,
        &[
            "[done", "[result", "[error", "[cancelled",
            "[archived:", "[archived",
            "[snooze", "[origin:tg",
        ],
    );
    collapse_whitespace(&cleaned)
}
```

复用既有 `remove_bracketed_segments` + `collapse_whitespace` 私有底层
helper — 与 `strip_done_markers` / `strip_archive_markers` 同模板，新
prefix 集合是「dup-specific 不可继承 markers」。

5 个 unit tests 覆盖：
1. 剥 [done] / [result:]
2. 剥 [snooze:] / [origin:tg:]
3. 保 [every:] / [reminderMin:] / [pinned] / [silent] / [blockedBy:] / #tag
4. 剥 [error:] / [cancelled:] / [archived:]
5. clean pending body 是 noop（幂等性）

#### 纯 formatter `format_dup_reply`

```rust
pub fn format_dup_reply(src_title: &str, new_title: &str) -> String {
    format!(
        "📑 已复制「{}」→「{}」\n· 继承 schedule / markers / tags / priority / due\n· 剥终态 markers（done / result / snooze / origin 等）",
        src_title.trim(),
        new_title.trim(),
    )
}
```

注脚显示 "继承" + "剥" 两行让 owner 看清楚副本得到什么 / 丢了什么 —
比 silent success 更 audit 友好。

5 个 unit tests：parser（含空 title）、formatter（src + new title 显示、
注脚双行、unique-filename `_N` 后缀场景）。

### `src-tauri/src/telegram/bot.rs`

Handler 紧贴 Peek 之后：

1. resolve 与 /show / /peek 同三层
2. 取 view 拿 raw_description
3. `parse_task_header` 抽 priority / due / body
4. `strip_for_dup(body)` 去终态 markers

## Key design decisions

- **剥 `[origin:tg:...]`**：副本是新 instance — origin chain 不该误指
  到源 task 的 chat id。后续 handler 可选择重新打 origin marker（当前
  iter 不引该 side-step，保留极简语义）
- **`(副本)` 后缀而非数字编号**：可读性优先。memory_edit unique-
  filename 已自动加 `_N` 处理冲突
- **priority 继承而非默认 P3**：TODO 原 wording 说 "新 P3 实例"，但 P3
  默认会让"重复 dup P7 高优任务"变成 P3 反向降级 — 用户预期更可能是
  "保留优先级"。如果 owner 想强制 P3，可走 `/pri <new title> 3` 一步
- **due 继承**：与 priority 同理。owner 想清 due 走 `/edit_due <new>
  clear` 一步。继承让 "克隆模板" 场景一步到位
- **不引 detail.md 复制**：第一版本范围内 detail.md 不复制 — 那是
  task 实例运行后的产物（LLM 写过的进度笔记），副本应回空白起跑。如
  果 owner 真想复制 detail content，走 detail.md 编辑器 ⌘A → ⌘C → 切
  新 task → 粘 — 不必为低频场景引复杂度
- **使用既有 `task_create` 而非新写 path**：让 unique-filename / memory_
  edit history / SQLite mirror / decision_log 等所有 side-effect 自动
  跟上 — TG /dup 与 PanelTasks「+ 新建」走同一后端
- **10 个 unit tests pin 真实行为**（满足 GOAL.md "meaningful tests
  only"）：
  - strip_for_dup 5 个：路径覆盖（终态剥 / 不可继承剥 / 可继承保 / 多
    路径剥 / 幂等）
  - parse + format 5 个：parser title 捕获 / 空 title / formatter 双标
    题显 / 注脚双行 / unique-filename `_N` 透显
