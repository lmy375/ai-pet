# 任务队列 — 开发计划

> 对应需求（来自 docs/TODO.md「已确认」）：
> 任务队列：面板支持创建/查看任务，宠物按优先级与截止时间自动取单执行并回流结果。

## 目标

让用户能在面板里**结构化创建任务**（标题 / 描述 / 优先级 / 截止时间），并看到一个按"宠物会优先做哪个"排好序的列表。宠物侧在 proactive 循环里照常通过 `butler_tasks_hint` 看到这些任务，结果通过现有的 `butler_history` + `[done]` / `[error: ...]` 标记回流。

不是从零造一套队列基础设施 —— 而是给现有的 `butler_tasks` 内存类目叠一层结构化 header 与排序逻辑，**兼容老条目**（无 header 的条目按已有行为对待）。

## 非目标

- 不引入新的存储后端 / 状态机 — `butler_tasks` 内存类目继续是真相源。
- 不做实时执行 / 调度独立线程 — 宠物仍在 proactive turn 中按需取单。
- 不替代 `[once: ...]` / `[every: HH:MM]` 现有 schedule 前缀 — 它们与新 `[task pri=...]` 正交，可同时出现。
- 不在本轮里重写 prompt 的 `butler_tasks_hint` 排序（保持现有「到期 → 最早委托」语义不变，避免回归）。新优先级排序先服务面板侧，后续迭代再视效果决定要不要回流到 prompt。

## 设计

### Header 约定

新增"任务"前缀 `[task pri=N due=YYYY-MM-DDThh:mm]`，可与 `[once:]` / `[every:]` / `[done]` / `[error:]` 共存：

```
[task pri=3 due=2026-05-05T18:00] 把 ~/Downloads 里的图片按月份归档
```

- `pri` ∈ 0..=9，**数值越大越紧急**。默认 0。
- `due` 可选，本地时区，分钟精度。无 due 的任务永远不"过期"，但优先级仍生效。
- header 失败解析（格式坏 / 越界 / 时间无效）→ `None`，回退为"无 header 任务"，与历史条目同等对待。

### Tauri 命令

```rust
#[tauri::command]
fn task_create(title: String, body: String, priority: u8, due: Option<String>) -> Result<String, String>
// 写入 butler_tasks 类目；header 自动拼装。返回新任务的 detail_path。

#[tauri::command]
fn task_list() -> Vec<TaskView>
// 读 butler_tasks → 解析 header → 过滤 [done]、按队列序排序后返回。

struct TaskView {
    title: String,
    body: String,           // 去掉 header 后的纯描述
    priority: u8,
    due: Option<String>,    // ISO，前端按本地时区渲染
    status: String,         // "pending" | "done" | "error"
    error_message: Option<String>,
    created_at: String,
    updated_at: String,
}
```

### 排序比较器（pure）

输入：`a, b: &TaskView`，`now: NaiveDateTime`。

返回：`Ordering`，"宠物应优先处理 a" → `Less`。

层级：

1. **状态优先**：error > pending > done（done 默认不返给面板，过滤掉；保留比较逻辑便于未来"显示已完成"开关）。
2. **过期优先**：有 due 且 due ≤ now 的排前；过期程度越久越前。
3. **优先级降序**：pri 大的排前。
4. **due 升序**：早到期的排前；无 due 视作 +∞。
5. **created_at 升序**：稳定 tie-break。

### 状态判定（pure）

- 描述含 `[error` → `error`（带消息：括号内文字）
- 描述含 `[done]`（独立 token，避免误命中 "done" 单词）→ `done`
- 否则 → `pending`

### 与 proactive prompt 的关系（本轮）

不改 `format_butler_tasks_block`。新建任务的 header 文本会被 `BUTLER_TASKS_HINT_DESC_CHARS=100` 截到 prompt 里 —— 让 LLM 看到 `[task pri=3 due=...]`，理解为优先级标记并自然地优先处理。模型不必"硬解析"，文本本身已是有效信号。
