# TG bot 加 `/pinned_drop_7d` 命令（iter #560）

## Background

正向「关注度」audit 已齐：/pinned（当前钉）/ /peek_pinned（紧凑）/
/random_pinned（抽一）/ PanelMemory 📊 7d chip（new growth）。反向
audit 缺：「owner 最近放下了哪几条」— pin → unpin 切换信号。这是
priority 管理是否健康循环的指标（光 pin 不 unpin → backlog 累积）。

## Best-effort detection algorithm

数据局限：backend 没显式 pin/unpin event log。butler_history.log 记
所有 `memory_edit` 后的 description snippet（80 字截断），所以我们
能 inferred-detect 但不严格：

```
candidates = title ∈ current_unpinned_views        (live state from views)
           ∧ ∃ butler_history line in last 7 days  (history scan)
             where snippet contains "[pinned]"     (= was-pinned sighting)
```

= 当前不带 [pinned] 但 7d 内曾被 update 写到含 [pinned] 的 task。

## Changes

按 6+ surface 同步：

1. Enum `PinnedDrop7d`（无参；紧贴 `CatGrowth7d`）
2. name / parser / no-args title chain / registry en+zh / help-detail /
   help-table / 三份 drift lists

### `format_pinned_drop_7d_reply(rows: &[(title, ts_label)]) -> String`

pure — caller 已 set-intersect + sort by ts desc + format ts label
`MM-DD HH:MM`：

- 空 rows → 教学指 /pinned（当前钉）+ /touched_thisweek（本周全谱）
- 非空 → header「📌💤 近 7 天 unpin 候选 N 条（pin → 当前 unpinned）」
  + 每行 `· 「<title>」 · 最后 [pinned] 见于 MM-DD HH:MM`
- cap 8（与 /find_speech 系列同）— 超出尾「还有 N 条候选」

### Handler (bot.rs)

紧贴 CatGrowth7d：

```rust
TgCommand::PinnedDrop7d => {
    let views = read_tg_chat_task_views(chat_id.0);
    let cutoff = chrono::Local::now() - chrono::Duration::days(7);
    let current_unpinned: HashSet<String> = views.iter()
        .filter(|v| !v.pinned).map(|v| v.title.clone()).collect();
    let mut latest_pin_ts: HashMap<String, DateTime<Local>> = HashMap::new();
    let content = crate::butler_history::read_history_content().await;
    for line in content.lines() {
        // parse ts_str + body; body split " :: " → head + snippet
        // if snippet.contains("[pinned]") && title ∈ current_unpinned
        // → update latest_pin_ts.entry(title).max(ts_local)
    }
    let mut with_dt: Vec<_> = latest_pin_ts.into_iter().collect();
    with_dt.sort_by(|a, b| b.1.cmp(&a.1));
    let rows: Vec<(String, String)> = with_dt.into_iter()
        .map(|(title, ts)| (title, ts.format("%m-%d %H:%M").to_string()))
        .collect();
    format_pinned_drop_7d_reply(&rows)
}
```

## Key design decisions

- **「snippet 含 [pinned]」探测**：因为 [pinned] marker 是 description
  内 ASCII token，按 contains 检查即可。但 80 字截断时若 [pinned] 在
  末尾被砍 → false neg（snippet 无 [pinned] 但实际 task 仍 pinned）。
  我们不计 truncated-no-pinned 为"未 pin"，仅记 contains-pinned 为
  "曾 pin"sighting。这样 false neg 偏向"漏报候选"（保守），不会"误
  报候选"（仍 pinned 被列）
- **「current_unpinned」live filter**：候选必须当前 not pinned。这样
  pin → unpin → repin 在 7d 内的 task 不会被列（最终态仍 pinned）—
  避免 owner 看「咦明明还钉着为啥也在 unpin 候选」
- **「title in current views」隐含**：current_unpinned 来自 views 已
  按 chat scope 过滤；title 不在 views 的 history 行被忽略 — 跨聊天
  泄漏避免
- **history line parse**：`<ts> <body>` 用首 ' ' split；body 用 ` :: `
  split snippet。`head` = `<action> <title>`，按首 ' ' 分；title 含空
  格也安全因后面整段都是 title 直到 ` :: `
- **ts desc sort + cap 8**：最近被取消的在上 — owner 想看「我刚 unpin
  了啥」；cap 8 与 /find_speech 系列保一致
- **caveats in help-detail**：明示 80 字截断局限 + history retention
  局限 + 「曾 pinned 后被 update 写了但 marker 留着」的混淆 — 让 owner
  知道这是 inferred audit 非严格
- **3 unit tests**：parser（含尾部多余 args）+ empty fallback + multi
  row + cap 8 + truncation tail hint
