## 改动

### `src/components/panel/PanelTasks.tsx`

#### handler

复用既有 `runBulk(label, predicate, skipReason, op)` 抽象 —— 与 `handleBulkRetry` 同模板：

```ts
const handleBulkPin = useCallback(async () => {
  await runBulk("钉住", (t) => !t.pinned, "已钉住", async (title) => {
    await invoke<void>("task_set_pinned", { title, pinned: true });
  });
}, [runBulk]);

const handleBulkUnpin = useCallback(async () => {
  await runBulk("取消钉住", (t) => !!t.pinned, "未钉住", async (title) => {
    await invoke<void>("task_set_pinned", { title, pinned: false });
  });
}, [runBulk]);
```

#### 工具栏 2 个按钮

在「改 tags」之后、「复制标题」之前插入：

```tsx
<button onClick={handleBulkPin} title="批量钉住（已钉住的跳过）">📌 钉住</button>
<button onClick={handleBulkUnpin} title="批量取消钉住（未钉住的跳过）">📌 取消钉</button>
```

## 关键设计

- **复用 `runBulk` 抽象**：与既有 ✓ 标 done / 取消 / 改优先级 / 改 due / 改 tags / 重试 共享同一份 progress 反馈（"钉住 N 条 ✓ · 跳过 M 条（已钉住）"）+ 同一 `bulkBusy` lock + 同一 reload 路径。新功能 0 行 boilerplate。
- **2 个按钮而非 toggle**：选中任务可能 pin / unpin 状态混杂；toggle 语义在混合集上是噪音（"切换" → 谁切换到啥？）。分立 `📌 钉住` 与 `📌 取消钉` 两个动作 + predicate 跳过 noop，让 owner 明确意图。
- **predicate skip 非 backend skip**：后端 `task_set_pinned` 已 strip-before-write 幂等。predicate 这里只是为了 toast 文案能告诉 owner "我跳过了 N 条已钉住"，而不是浪费一轮 IO 也不告知。
- **不加输入面板**：pin 是 boolean 无参数 —— 与既有 cancel reason / due time / priority 等"需要二级输入"的 bulk action 不同。点按钮即执行，零步骤。
- **位置在 tags 与复制之间**：保 bulk 工具栏的"破坏性 / 修改性 / 复制 / 选择"四段分组完整 —— pin 属"修改性"（与 done / cancel / priority / due / tags 同段）。

## 不做

- **不接键盘快捷键**：bulk 工具栏目前都没快捷键（只 row-level d / r / Delete 等）。单独给 pin 加 `p` / `P` 可能与未来扩展冲突，留待整体快捷键梳理时再补。
- **不做"已选中含 pinned 和 unpinned 混合时智能 toggle"**：mixed-state toggle 是 anti-pattern（owner 期望"全部 pin / 全部 unpin"而非"翻转每条"）；分立按钮更可控。
- **不写测试**：纯前端 onClick + `runBulk` 调用，无可单测的纯函数；既有 `runBulk` 路径已被 ✓ 标 done / 重试 / 取消 等多条 callsite 覆盖（同 progress / skip / reload 流），新 callsite 共享同一行为。
