# TG /tasks 查询命令 — 开发计划

> 对应需求（来自 docs/TODO.md）：
> TG /tasks 查询：用户在 TG 发 `/tasks` 列出自己派单的任务清单 + 当前状态，不必切回桌面也能看进度。

## 目标

让 Telegram 用户在不切回桌面的前提下查看"自己通过 TG 派单的任务"当前状态。
新加一条 TG 命令：

- `/tasks` — 列出当前 chat_id 派出的全部任务，按状态分组（进行中 / 已完成 / 已失败 / 已取消）。

只看"本会话派出"的任务（按 `[origin:tg:<chat_id>]` 过滤），原因：
- TG 用户视角：他想看的是"我刚才派给宠物那一坨"，不是宠物全部 butler_tasks（包含 LLM 自建 / 桌面派单 / 周报等）；
- 隐私 / 噪音：群组场景里 A 派的任务不该在 B 的 /tasks 里冒出来；
- 与 watcher 的"完成回传只发回 origin chat"语义对齐。

## 非目标

- 不做分页 / 不做 `since:` 过滤 —— TG 派单条数通常 < 20，单消息即可塞下。如真触顶（TG 单条 4096 字节）再做截断+提示。
- 不做"查特定任务详情"——detail.md 在 TG 里展示也很难看，要细节让用户回桌面。
- 不做命令别名（`/list` 之类）—— 只暴露 `/tasks` 一条，避免与未来命令冲突。
- 不区分 user 与 group chat（Telegram 视角下都是 chat_id 比对即可）。
- 不写 README —— 这是 TG 命令矩阵的延伸（cancel / retry / tasks 三件套），README "Telegram 派单"段落已隐含闭环，不必为单条命令再加亮点行。

## 设计

### 解析

`telegram::commands::TgCommand` 加 variant：

```rust
pub enum TgCommand {
    Cancel { title: String },
    Retry { title: String },
    Tasks,
    Unknown { name: String },
}
```

`parse_tg_command` 增加 `"tasks" => Some(TgCommand::Tasks)` 分支。
没有参数（`/tasks abc` 多余尾部仍命中 Tasks，忽略尾部；与 cancel/retry "title 必填"对偶但语义相反）。
未知命令文案补一条 `/tasks` 进可用清单。

### 文案 helper（pure）

新增 `format_tasks_list(views: &[TaskView]) -> String`。输入是已过滤、已按
`compare_for_queue` 排序好的 TaskView 列表（按 origin chat_id 过滤后）。

输出格式：

```
📋 你的任务（共 N 条）

进行中（K）
⏳ P3 整理 Downloads — 截至 5/5 18:00
⏳ P1 喝水

已完成（K）
✅ 写周报

已失败（K）
⚠️ 跑步 — 下雨了

已取消（K）
🚫 学习 Rust — 改主意了
```

规则：
- 空列表 → `"📋 你的任务清单是空的，想派点啥？"`（与"加到队列里了"风格呼应）。
- 每行 `<emoji> [P{pri}] <title> [— <suffix>]`：
  - 进行中：`⏳`，附 due（若有）格式 `截至 M/D HH:MM`，无 due 不加 suffix。
  - 已完成：`✅`，附 result（若有）；result 长度过 40 char 截断+省略号，避免单行炸开。
  - 已失败：`⚠️`，附 error_message（若有），同样 40 char 截断。
  - 已取消：`🚫`，附 cancellation reason（同上）。
- 优先级在标题左边只在 pri > 0 时展示（默认 0 不显眼，省字数）。
- 段落间空行；空段落（K=0）整段省略，避免一堆 `已失败（0）`。
- 末尾不加签名 / 帮助文字（一旦消息塞 TG 单条上限 4096，再考虑折叠）。

整体 ≤ 4096 byte 防御：在最末追加前测算长度，超出就在末尾追加
`"…(列表过长，剩余 N 条请回桌面查看)"` —— 实战上几乎不会触发。

### IO 路径

`handle_tg_command` 加 `TgCommand::Tasks` 分支：

```rust
TgCommand::Tasks => format_tasks_for_chat(msg.chat.id.0)
```

`format_tasks_for_chat(chat_id: i64) -> String`（在 bot.rs 内）：
1. `memory::memory_list(Some("butler_tasks"))` 拿 index
2. 遍历 `cat.items`，对每条 parse origin；origin 是 `Tg(chat_id)` 时收入候选
3. 对候选用 `commands::task::build_task_view`（如果是私有的就要把它公开） → 拿到 `TaskView`
4. `compare_for_queue` 排序
5. 调 `format_tasks_list` 拿字符串
6. 错误（memory_list 失败 / 类目缺失）：返回 "📋 你的任务清单是空的…"，不抛错给用户

build_task_view 当前是 `commands/task.rs` 内私有。要公开。命名 `build_task_view`
保持不变，加 `pub(crate)` 即可（不暴露给前端 invoke 协议）。

### 测试

`commands.rs`（pure，单测好写）：
- `parse_tg_command("/tasks")` → `Some(TgCommand::Tasks)`
- `parse_tg_command("/TASKS")` → `Some(TgCommand::Tasks)`（大小写不敏感）
- `parse_tg_command("/tasks something")` → `Some(TgCommand::Tasks)`（尾参数被忽略）
- `format_unknown_command` 现在应包含 `/tasks`（更新现有断言）
- `TgCommand::Tasks::name()` 返回 `"tasks"`
- `TgCommand::Tasks::title()` 返回 `""`

`format_tasks_list`（pure）：
- 空列表 → 提示句
- 单条 pending → 包含 `⏳`、标题、due 文本（若有 due）
- 多状态混合 → 4 个分组按"进行中 → 已完成 → 已失败 → 已取消"排
- 空段落不出现（验证不出现 `已失败（0）` 字样）
- result/error 超长被截断（含 `…`）
- 优先级 0 不渲染 P 前缀；>0 渲染 `P3` 这种

bot.rs 的 `format_tasks_for_chat` 涉及 IO，**不做** memory mock —— 单测意义有限，
通过手动 TG 测试覆盖。

## 复用清单

- `commands::task::build_task_view`（拟由 private 提升 `pub(crate)`）
- `task_queue::{parse_task_origin, TaskOrigin, compare_for_queue, TaskStatus, TaskView}`
- `memory::memory_list` —— 直接读 butler_tasks
- `telegram::commands::format_*` 现有命令成功 / 失败 helpers 不复用，本命令是查询不是动作。

## 待用户裁定的开放问题

- 是否需要在 `/tasks` 输出里也展示 LLM 自建 / 桌面派单的任务？本轮选**否**（只看自己派的）。如反馈"我想看全量"再加 `/tasks all`。
- 是否需要对 `compare_for_queue` 在 TG 输出场景做调整？本轮选**否**（沿用面板顺序，进行中→完成→取消）。
- TG 群组场景下，多人各自 `/tasks` 互不干扰 —— 这正是按 `chat_id` 过滤的副产品（群里 chat_id 相同则会互看；要更严要按 `from.id` 过滤，但进入"用户身份"边界，本轮不做）。
