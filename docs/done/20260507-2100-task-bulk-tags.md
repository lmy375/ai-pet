# 任务列表批量改 tags — 开发计划

> 对应需求（来自 docs/TODO.md）：
> 任务列表批量改 tags：select 多条 → 输 `+tag1 -tag2` → 批量 add/remove tag，让 LLM 给一堆任务统一打或卸标签时不必一条一条改。

## 目标

PanelTasks 已支持单任务 tag chip 切换 + 批量取消 / 改优先级 / 改 due。
但批量改 tag 缺失：LLM 一次塞 10 条任务都没打 tag、或者所有 tag 都错了
要重命名时，要逐条点 chip。本轮加一个批量 tag 操作输入：用户输
`+tag1 -tag2 +工作` 单行 → backend 解析 → 对每条选中任务在 description
里加 / 删对应 tag。

## 非目标

- 不做 GUI tag picker 多选（chip 选 + 操作）—— 文本语法 `+/-` 紧凑直观，
  桌面用户输入快。
- 不做 tag rename（renames-all 操作）—— 删 + 加可达，无需新 op 类型。
- 不做"清空所有 tag" 操作 —— 用户能输 `-tag1 -tag2 ...` 显式列出；隐式
  全删容易误操作。

## 设计

### 解析输入

`parse_tag_ops(input: &str) -> Result<Vec<TagOp>, String>`：
- token 用空白切；每 token 必须以 `+` 或 `-` 开头
- 移除前缀后剩余为 tag 名；为空 → 整体 Err
- 重复 op 去重保留首个（`+a +a` → 一次 add）
- 互斥冲突（`+a -a` 同次输入）→ Err，让用户重输；不"先加后删 = 净删"
  这种潜规则

```rust
pub enum TagOp {
    Add(String),
    Remove(String),
}
```

### 应用到 description

`apply_tag_ops(description: &str, ops: &[TagOp]) -> String`：
- Add：if `parse_task_tags(description)` 不含该 tag → append ` #tag` 到末尾
- Remove：扫 description 中所有 `#tag` 出现位置（用与 `parse_task_tags`
  一致的边界判定）→ 删该 token + 前导空格
- 多 op 顺序应用（互斥已被 parse_tag_ops 拒绝）

### Tauri 命令

`task_set_tags(title, ops_input: &str)`：
1. parse_tag_ops 校验
2. find_butler_task → ok 拿 description
3. apply_tag_ops 算新 description
4. memory::memory_edit("update", ...) 写回
5. 推 decision log

### 前端

bulk toolbar 新增 "改 tags" 按钮，与既有 cancel / priority / due 同套
模式：点击展开 sub-panel → 输入 `+/-tagname` 单行 → 确认。

提示文案 placeholder：`+tag1 -tag2 +工作`，title hover 解释 `+加 / -删，
空白分隔；同次输入不能同时 +X -X`。

### 测试

后端单测：
- parse_tag_ops 解析正常 / 空 / 单纯 +/- / 中文 tag / 重复去重
- parse_tag_ops 拒绝 `+a -a` 互斥
- parse_tag_ops 拒绝 `+` 后无名
- apply_tag_ops 加新 tag 末尾
- apply_tag_ops 删除已有 tag
- apply_tag_ops 加重复 tag 不重复
- apply_tag_ops 删不存在 tag noop
