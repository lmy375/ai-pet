# Telegram /task 命令支持快速建任务 — 开发计划

> 对应需求（来自 docs/TODO.md）：
> Telegram /task 命令支持快速建任务：现在 /task 只列任务，加 `/task new <title>` 直接走和 task_create 同源的入口，复用面板创建逻辑。

## 目标

TG bot 已有 `/tasks`（列）/ `/cancel` / `/retry`，但**新建**任务必须靠
LLM 自由对话（"帮我整理 Downloads"→ LLM 决定调 task_create 工具）。这条
路径在网络抖动 / LLM 偶尔走神的情况下会变成"我说了它没记"。本轮加显式
的 `/task <title>` 命令，**不经 LLM**，直接落到 butler_tasks，给用户一条
可靠 fallback。

## 非目标

- 不改 LLM 端 `task_create` 工具 —— 让 LLM 在自由对话里仍能派任务。
- 不在 TG 上做 priority / due / body 输入 UI —— 那需要多步交互（/task
  → "请输入优先级" → ...），价值低 & 易卡住；想精细派单的用户回桌面板。
  TG 路径只支持 `/task <title>`，`priority=3`（与桌面默认对齐），无 due，
  无 body。
- 不做 `/task new <title>` —— 单数 `/task` 已和复数 `/tasks` 在 TG 客户端
  自动补全里区分得很清；再加 `new` 子命令是徒增噪音。

## 设计

### 命令名字

- `/tasks`（复数）= 列表，已存在
- `/task <title>`（单数）= 创建一条，新增

TG 客户端的命令补全会同时显示两条。文案显式告诉用户"单数 = 创建，复数 = 列表"。

### 解析（pure，commands.rs）

```rust
pub enum TgCommand {
    // 既有...
    Task { title: String },
}
```

`parse_tg_command` 在 match 表中加 `"task" => Some(TgCommand::Task { title })`
分支。空 title → handler 走 `format_missing_argument("task")`，与 cancel /
retry 同语义。

### 处理（IO，bot.rs）

`handle_tg_command` 的 match 加 `TgCommand::Task` 分支：

1. trim title；空 → format_missing_argument
2. 拼 `TaskHeader { priority: 3, due: None, body: "".into() }` →
   `format_task_description` → `append_origin_marker(&desc, &TaskOrigin::Tg(chat_id))`
3. 调 `memory::memory_edit("create", "butler_tasks", title, Some(desc), Some(""))`
4. 成功 → 回 `format_task_created_success(title)`；失败 → format_command_error

为什么不复用 `task_create_tool::build_description`：那是 LLM 工具私用的
`TaskCreateArgs` 类型；在 bot.rs 直接拼三行更清楚，且已暴露的 task_queue
原语（format_task_description / append_origin_marker）就是为这种复用准备的。

### 文案（pure，commands.rs）

```rust
/// `/task <title>` 成功反馈。强调"已入队 + 默认 P3 + 怎么改细节"。
pub fn format_task_created_success(title: &str) -> String {
    format!(
        "✅ 已加到队列「{}」(P3)\n用 /tasks 查看，/cancel {} 撤回；想调优先级 / 截止时间请回桌面板",
        title.trim(),
        title.trim()
    )
}
```

### help 文本

`format_help_text` 加一行 `/task <title>  —  把单条任务塞进队列（默认 P3，
无 due；细节回桌面板调）`。

### 测试

`commands.rs`：
- parse_tg_command 识别 `/task xxx` → `TgCommand::Task { title: "xxx" }`
- parse_tg_command 识别 `/task`（无参）→ `Task { title: "" }`，由 handler
  走 missing-argument
- name() 返回 "task"
- format_task_created_success 包含 title + P3 + /tasks + /cancel 关键字

`bot.rs`：handler 路径是 IO，不直接单测；但既有 dispatcher 测试模式不变。

## 阶段划分

| **M2** | format_task_created_success + help 文本一行 + 单测 |
| **M3** | bot.rs handle_tg_command 接入 + chat_id 取 origin + memory_edit |
| **M4** | cargo test + tsc + build + cleanup |

## 复用清单

- 既有 `format_task_description` / `append_origin_marker` / `TaskOrigin::Tg`
- 既有 `format_command_error` / `format_missing_argument`
- 既有 memory::memory_edit "create" 路径
- 既有 task watcher：origin marker 一旦写入，watcher 完成 / 失败时会自动回传
