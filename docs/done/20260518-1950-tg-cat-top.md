# TG `/cat_top [N]` 命令（iter #589）— capacity axis cat audit

## Background

近月加了 4 个 cat audit 命令（/cat_growth_7d / 30d / /cat_decay_7d /
30d）— 都是**活跃度 axis**（最近多动 / 多新）。缺**容量 axis** —
「我哪类知识累计最多」总量对比。

owner 想看：
- 哪些是「主力 cat」（item 多 → 重点维护）
- 哪些是「边缘 cat」（item 少 → 是否该 archive / consolidate）

## Changes

6-surface 同步：

1. Enum `CatTop { n: u32 }`（紧贴 `AuditSummary`）
2. name / parser (N clamp 1..=20，缺省 5) / title chain / registry en+zh
   / help-detail / help-table / 三份 drift lists
3. 新 pure formatter `format_cat_top_reply(rows, total_cats)`
4. Handler scan `memory_list(None)` + per-cat item count + sort desc +
   cap N。空 cat 跳过（0 count 不入 row）
5. /help_table cat 家族列表 + family detail map 加 /cat_top

## Output

```
📊 cat top N（按 items 总量 desc，共 M cat in index）：
· butler_tasks · 156 条
· decisions · 89 条
· general · 42 条
...
```

空 → 「memory index 内无 cat」+ 教学指 /cat_growth_7d。

## Key design decisions

- **「capacity axis 与 activity axis 正交」框架**：help-detail 明示
  差异 — owner 心智上区分「主力 cat (多)」vs「最近动的 cat (热)」。
  两 axis 命令 family 互补
- **空 cat 跳过**：与 /cat_decay_7d 同 — 0 count cat 不入 row 因为
  cap N 后会被砍 + 视觉噪音
- **count desc + key asc tie-break**：稳定输出便测
- **header 含 `共 M cat in index`**：让 owner 知 N=5 是 cap 还是真总
  数。M > N 时显示 cap 状态隐含
- **4 unit tests**：parser default + parser clamp（upper / lower）+
  empty fallback + multi-row（count desc 排序断言）
