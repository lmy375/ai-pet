# TG bot 加 `/peek_pinned` 命令（iter #533）

## Background

既有 pinned-related 命令：
- `/pinned` — 列所有 pinned task 标题（标题-only）
- `/peek <title>` — 单 task 一行紧凑视图（status + schedule + markers）
- `/pinned_due` — pinned 且有 due 的子集

但缺**「所有 pinned 一行紧凑批量」入口** — owner 想看「我钉的 N 条状
态都如何」时只能：
1. /pinned 看标题
2. 逐条 /peek <title> 看详情

或者 /tasks 全量 + 肉眼挑 pinned。本 iter 加 `/peek_pinned` 一步给所
有 pinned 的紧凑视图。

#### 纯 formatter `format_peek_pinned_reply`

```rust
pub fn format_peek_pinned_reply(views: &[TaskView]) -> String {
    if views.is_empty() {
        return "📌 暂无 pinned task。\n用 /pin <title> 钉一条任务...";
    }
    let mut out = format!("📌 {} 条 pinned：", views.len());
    for v in views {
        out.push('\n');
        // 复用单 task /peek formatter — 每行一致 schedule + markers +
        // 状态 emoji 渲染
        out.push_str(&format_peek_reply(&v.title, &v.raw_description, v.status));
    }
    out
}
```

复用既有 `format_peek_reply` — 单行格式与 /peek `<title>` 完全一致，
让 owner 切换两入口时心智无歧义。

### `src-tauri/src/telegram/bot.rs`

Handler 紧贴 Pinned 之后 — 同 chat-scope filter + 同 pinned filter，
仅 formatter 不同：

```rust
TgCommand::PeekPinned => {
    let views: Vec<TaskView> = read_tg_chat_task_views(chat_id.0)
        .into_iter()
        .filter(|v| v.pinned)
        .collect();
    format_peek_pinned_reply(&views)
}
```

## Key design decisions

- **复用 format_peek_reply 单行渲染**：保两 axis 一致 — owner 看
  /peek_pinned 的某行与单独 /peek <title> 的输出格式完全相同，心智复
  用
- **无 N cap**：pinned task 通常少（owner 钉自己最在意的几条，典型 < 10
  条）— cap 反而漏看
- **空集教学指 /pin**：让 owner 知道下一步如何创 pinned（与既有
  /touched_today 教学 /find 等同 pattern）
- **不写新 Schedule 解析**：复用 format_peek_reply 既有 schedule 检测
  限制（仅 raw 起始位置识别）— 与既有 /peek `<title>` 行为一致；改进
  schedule 检测应在 format_peek_reply 而非本 wrapper（避免行为分裂）
- **3 个 unit tests pin 真实行为**（满足 GOAL.md "meaningful tests
  only"）：parser（含尾部 token 容忍）+ 空集教学 + batch list（header
  + 每行 /peek format wiring 验证）
- **测试发现既有 schedule 检测限制**：format_peek_reply 仅在 raw 起始
  位置识别 `[every|once|deadline:]`，而 production task 通常起
  `[task pri=N]` header。已在 test 里注释，作为既有 /peek 已知限制保
  留；不在本 iter 范围内改进（避免 scope creep）
