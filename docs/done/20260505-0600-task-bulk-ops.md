# 任务批量操作 — 开发计划

> 对应需求（来自 docs/TODO.md）：
> 任务批量操作：面板支持多选任务一次取消 / 重试 / 改优先级，长队列下减少逐条点击。

## 目标

「任务」面板长队列下逐条 retry / cancel / 调优先级太琐碎。本轮加：
- 每行加 checkbox，选中后顶部出现"批量操作"工具栏
- 工具栏支持：批量重试 / 批量取消（共享 reason）/ 批量改优先级 / 全选 / 取消选择
- 每个动作循环调对应单条命令；非法状态（如 retry 一个 done 任务）自动 skip 并在结果中汇总

## 非目标

- 不做后端 bulk RPC —— 现 5-10 选择规模下，N 个 single-op 的 IO 开销可以忽略。
  真要加 bulk-RPC 可以未来再做。
- 不做"批量改截止时间"——批量改 due 实际语义模糊（同一 due 给多条意味着一刀切，
  几乎不会想这么做）。
- 不做拖拽多选 / 区间选择（shift-click）—— checkbox 简单且足够。
- 不做 keyboard shortcut（cmd+a 全选等）—— UI 按钮一键已够用。
- 不写 README —— 体验补强，不是新独立亮点。

## 设计

### 后端

新加 `task_set_priority(title, priority)` Tauri 命令：

```rust
#[tauri::command]
pub fn task_set_priority(title: String, priority: u8) -> Result<(), String>;
```

实现：
1. 校验 priority ≤ TASK_PRIORITY_MAX（9），否则 Err
2. `find_butler_task(title)` → MemoryItem
3. parse_task_header(&description)：
   - Some(h) → 创建 new header `{ priority, due: h.due, body: h.body }`
   - None（legacy 无 header） → `{ priority, due: None, body: description.trim() }`
4. format_task_description(&new_header) → new_desc
5. memory_edit("update", "butler_tasks", title, Some(new_desc), None)

不推 decision_log（与已有 retry/cancel 不同 —— 改优先级是日常 UX 调整，进
log 是噪音）。

不需要 `_inner` 拆分（暂无 TG / 非-Tauri 路径调用需求）。

### 前端

`PanelTasks.tsx` 状态新增：

- `selected: Set<string>` — 选中的 task title 集合（按 title 索引；同 title
  重名的边缘情况理论存在但 UI 也无法区分，沿用现有 retry/cancel 都按首条
  匹配的语义）
- `bulkBusy: boolean` — 批量执行中（禁用工具栏按钮）
- `bulkReason: string` — 批量取消的共享 reason
- `bulkPriority: number` — 批量改优先级的输入值
- `bulkAction: "cancel" | "priority" | null` — 当前展开的二级输入面板
- `bulkResultMsg: string` — 执行后的简短结果（"重试 5 条，跳过 1 条非 error"）

UI：

- 每行 header 左侧：现有 chevron 之前再加一个 checkbox。点 checkbox 不应
  触发 expand（`e.stopPropagation()`）。
- 队列上方：工具栏在 `selected.size > 0` 时出现：
  - `已选 N · 重试 · 取消 · 改优先级 · 全选 · 取消选择`
  - 全选 = 把当前 visibleTasks 都加入 selected（不仅当前页，但 visibleTasks
    已经是过滤后的可见集合，符合"我看到的全选"直觉）
  - 取消选择 = 清空 selected
  - 重试 / 取消 / 改优先级 → 各 entry point
- 二级面板（在工具栏下方折叠展开）：
  - 取消：`<input placeholder="取消原因（可留空）" /> [确认取消] [关闭]`
  - 改优先级：`<input type=number 0-9 /> [确认] [关闭]`
- 执行后：reload + 清 selected + 显示 `bulkResultMsg`（"重试 5 条 ✓ / 跳过
  1 条（非 error 状态）"），4-5s 后自动清掉。

### 客户端循环执行

- 重试：对每条 selected title，若 `tasks` 里对应状态 == "error" → invoke
  task_retry；否则跳过（计入 skipped 计数）。
- 取消：对每条 selected title，若状态 ∉ {done, cancelled} → invoke
  task_cancel(title, reason)；否则跳过。
- 改优先级：对每条 selected title 都调 task_set_priority(title, priority)；
  允许 done/cancelled（无害，priority 在终态下只影响显示）。

错误聚合：失败计数 + 末尾错误（避免 N 条都失败时刷 N 条 toast）。

### 测试

后端 `task_set_priority` 走的是已有 helpers (parse / format / memory_edit)，
逻辑层级薄；纯函数已经在 task_queue::tests 里钉牢。本轮**只加一条集成型
测试**：跑 `task_set_priority` 的 Result 校验路径（priority > 9 → Err），
其它路径要 mock memory_edit，成本不值。

前端无测试基础设施，靠 tsc + 手测。

## 待用户裁定的开放问题

- checkbox 是默认显示还是 toggle 进入"选择模式"再出现？本轮选**默认显示**
  （多个任务面板的标准模式；toggle 加一层心智成本）。
- 全选行为：visibleTasks（过滤后）vs tasks（全部）？本轮选 **visibleTasks**
  —— 用户期待"全选"= "把我当前能看到的全选上"。
- 取消的 reason 在批量场景共享 vs 每条独立？本轮选**共享**（独立等于退化成
  逐条点击）。
