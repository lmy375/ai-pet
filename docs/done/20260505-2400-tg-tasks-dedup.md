# TG /tasks 命令历史去重 — 开发计划

> 对应需求（来自 docs/TODO.md）：
> TG 命令历史去重：用户连续两次 `/tasks` 时只回最新一份；当前每次都全量发，列表大时刷屏。delta 比对（与上次同则改回简短"无变化"）。

## 目标

TG 用户连续按 `/tasks`（或在 watcher 通知"已完成"后好奇心追问"现在还剩啥"），
当前每次都全量回 N 行任务清单。如果两次之间没有任何任务变化，TG 历史里堆
两份相同的列表，刷屏 + 阅读疲劳。

本轮在 bot process 里加 per-chat 上次响应缓存，命令命中后比较新旧 body 字符串
完全相等 → 发简短"📋 任务清单没变" + 提示用户怎么强制刷新（其实就是发别的
命令再回来；或加内容到任务后会自然变化）。

## 非目标

- 不做"diff 渲染"（只展示变更条）—— 实现复杂度高，对 < 20 条小列表收益小；
  本轮就用整体相等比对足够。
- 不持久化跨进程（重启 bot 即缓存清空）—— 重启相当于"全新一次"，再发完整
  列表合理。
- 不写 README —— TG 命令体验微调。

## 设计

### 后端

`HandlerState` 加 `last_tasks_response: TokioMutex<HashMap<i64, String>>`：
键 chat_id，值 上次 `/tasks` 发出的完整 body。

在 `handle_tg_command` 的 `TgCommand::Tasks` 分支：
1. 拿当前 body = `format_tasks_for_chat(chat_id)`
2. 锁 cache 检查 prev = cache.get(chat_id)
3. 若 `prev == Some(&body)` → 回 `format_tasks_no_change()` 简短文案
4. 否则 → cache.insert + 发完整 body

`format_tasks_no_change()` 是 pure helper（在 `telegram::commands`）：

```rust
pub fn format_tasks_no_change() -> String {
    "📋 任务清单和上次一样，没有变化。".to_string()
}
```

为何是 fn 而不是常量：与 `format_help_text` / `format_command_success` 等
现有 formatter 保持一致的风格，未来加上 chat_id 参数 / 时间戳更易扩展。

### 测试

- `format_tasks_no_change` 返回字符串含"无变化"或"没有变化"的关键词
- dedup 逻辑涉 TokioMutex + state，不做集成测试（与 `record_proactive_outcome`
  等 IO 路径同等级"成本不值"）；纯逻辑边界由 cache HashMap 自身保证

## 阶段划分

| 阶段 | 范围 |
| --- | --- |
| **M1** | `format_tasks_no_change` pure helper + 单测 |
| **M2** | `HandlerState` 加 `last_tasks_response` + 在 start() 初始化 |
| **M3** | `handle_tg_command::Tasks` 分支调 dedup |
| **M4** | cargo test + cleanup |

## 复用清单

- 既有 `format_tasks_for_chat`（read butler_tasks → 渲染）
- 既有 `HandlerState` Arc 模板（与 session_messages 同 TokioMutex 模式）

## 待用户裁定的开放问题

- "无变化"文案是否带具体计数（如 "10 条任务无变化"）？本轮选**否**——
  保持最简，用户想看具体数字可手动等待 watcher 推送或别的命令。
