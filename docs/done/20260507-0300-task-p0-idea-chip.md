# 任务 priority 0 chip — 开发计划

> 对应需求（来自 docs/TODO.md）：
> 任务 priority 0 chip：现已有 P3+ 优先级派发；P0 是"想到就好，没急"的 idea 抽屉；加 chip "P0 N" 一键过滤，让低优先级灵感不淹没活动队列。

## 目标

PanelTasks 现在按 queue / due 排序时，P0 任务（"想到就先记下，没急"）
和 P3+ 活动任务混在一起。LLM / 用户记的 idea 越来越多，活动队列被"以
后再说"的低 priority 项稀释。本轮加一个 "💡 P0 N" chip：开启时只看
`priority === 0` 的任务，把它当 "idea 抽屉" 单独打开。

## 非目标

- 不动后端 priority 字段语义 —— 0..9 已存在，无需新建"idea"概念。
- 不与 dueFilter 互斥 —— 不同 axis（time 紧迫度 vs 重要度）；用户开
  P0+overdue 想看的是"想到了但一直没做的老 idea"，是有效组合。
- 不做 P9 / 高优 chip —— 高优任务靠默认排序就在顶部，不需要"过滤出来"。
- 不持久化 chip 选中态 —— 与既有 dueFilter / selectedTags 同语义，临时
  视角。

## 设计

### state

`p0Only: boolean` default false。与 selectedTags / dueFilter 同级，独立
state（不挤进 dueFilter enum，axis 不同）。

### 计数

`p0Count` useMemo 派生 tasks 全集，不被 search/tag/dueFilter 链上过滤
影响 —— 与既有 `dueTodayCount` / `overdueCount` 同模式：让用户即使在
其它 filter 下也能看到 idea 抽屉总数。

```ts
const p0Count = useMemo(
  () => tasks.filter((t) => t.priority === 0 && !isFinished(t.status)).length,
  [tasks],
);
```

只数活动态（与 dueTodayCount 同语义 — 已结束 P0 不在 idea 抽屉里，他们
是"已经做掉的 idea"，回头看靠 showFinished）。

### 过滤合成

filteredTasks 链已有 status / dueFilter / search / tag 四段；新增：
```ts
.filter((t) => !p0Only || t.priority === 0)
```

放在 dueFilter 之后、search 之前。原因：dueFilter 的 active 段已经会
做 status 检查，p0 在它之后只关心 priority 字段，链路清晰。

### filtersActive

加 `p0Only` 一并视作 active 信号。

### UI chip

在 DueChip 同一行（`tagFilterRow` style 那行）末尾插入 P0 chip：
- 配色：slate gray（`#e2e8f0` bg / `#475569` fg）— "低能量、慢节奏"的
  视觉语义，区别于橙色（今日急）/红色（逾期）。
- 文案：`💡 P0 N`，💡 emoji 强调"灵感"含义；selected 时填充深色。

count 为 0 时 chip 不渲染（同 DueChip 模式，无事可做的日子不占位）。
